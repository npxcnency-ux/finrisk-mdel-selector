# finrisk-model-selector

财务风控场景的 ML 算法选型 Skill，适用于 Claude Code。

## 解决什么问题

财务风控建模的核心矛盾是 **预测效果 vs 可解释性 vs 实施成本**。面对"用什么模型"这个问题，选错路径的代价很高：

- 异常率 < 1% 时直接上 XGBoost → 模型全判正常，Recall 接近 0
- 用未确认的"疑似"标签训练 → 标签偏差被模型放大
- 追求 Recall 不考虑审核容量 → 审核员疲劳，模型形同虚设
- 随机 split 数据集 → 时间泄露，回测完美上线骤降
- 特征只有 3-4 个就硬上复杂模型 → 不如简单规则引擎

本 Skill 提供一套 **Step 0-4 决策框架**，从业务目标出发，经数据就绪度、标签状态、数据量与不平衡联合评估、可解释性要求，给出结构化的选型建议。

## 适用范围

**覆盖**：表格数据的批量异常检测 / 风险评分 — 费用报销审核、差旅异常、供应商付款风控、反欺诈稽核等。

**不覆盖**：实时流处理（Flink/Kafka Streams）、图像类欺诈检测（发票 OCR）、NLP 合同审计、信贷授信评分卡（A/B/C 卡体系）。

## 选型流程（Step 0-4）

```
Step 0: 业务目标与运营约束 → 误报/漏报权衡、审核容量、监管要求
Step 1: 数据就绪度评估 → 特征数量/质量、缺失率、时间字段可靠性
Step 2: 标签状态 → 有标签 / 弱标签 / 无标签，决定监督/半监督/无监督路径
Step 3: 数据量与不平衡联合评估 → 有效异常样本数决定模型复杂度上限
Step 4: 可解释性要求 → 区分合规团队 vs 被标记员工两类受众
```

## 输出格式

每次推荐包含 7 个组件：

1. **业务目标确认** — 误报/漏报权衡、运营容量、监管要求
2. **数据就绪度评估** — 特征数量/质量、缺失率、高基数特征
3. **推荐算法** — 主推 + 备选及选择理由
4. **不平衡处理方案** — 根据异常比例选策略
5. **解释性方案** — 区分合规团队 vs 被标记员工的不同需求
6. **评估指标** — PR-AUC / Precision@K / Recall，说明为什么
7. **下一步行动** — 数据准备 → 基线 → 迭代路径

## 关键设计决策

| 决策 | 理由 |
|------|------|
| Step 0 业务目标放首位 | 误报/漏报权衡决定评估指标，运营容量决定阈值天花板，最后考虑会导致前面选择推倒重来 |
| Step 1 新增数据就绪度 | 特征 < 5 个或缺失率 > 50% 时，ML 模型不如规则引擎，先评估避免浪费 |
| Step 3 合并数据量与不平衡 | 有效异常样本数（总量 × 异常率）才是真正的决策变量 |
| 异常率 < 1% 时不直接纯监督 | 核心反直觉规则：先 Isolation Forest 扩召回，再监督精排 |
| 区分两类可解释性受众 | 给合规排序（效果优先）≠ 向员工解释（强解释性），需不同模型 |
| 三层生产架构（规则 → 无监督 → 有监督） | 支持冷启动逐步演进，不强依赖标签即可上线 |
| 运营容量先于指标优化 | 先问"每天能审多少条"再定阈值，避免预警淹没审核团队 |

## 触发条件

当用户提到以下关键词或描述相关场景时自动触发：

- 中文：选什么模型、用什么算法、有/没有标签、数据不平衡、需要可解释、财务风控建模、异常检测建模、评分卡、冷启动、报销审核
- 英文：model selection, fraud detection, expense anomaly, imbalanced dataset, anomaly detection, risk scoring, scorecard, AML, cold start

## 评估

`evals/evals.json` 包含 5 个测试场景，覆盖：

| # | 场景 | 核心考察点 |
|---|------|-----------|
| 1 | 差旅报销 8 万条，异常 0.75% | 业务目标确认、严重不平衡两阶段策略、PR-AUC 选择 |
| 2 | 供应商付款 15 万条，无标签 | 业务目标澄清、数据就绪度评估、无监督路径、高基数特征 |
| 3 | 信用卡 50 万条，双受众解释需求 | 业务目标确认、区分两类可解释性、不能只靠 SHAP |
| 4 | 理赔 2 万条，180 条疑似标签 | 业务目标关联、弱标签识别、半监督路径 |
| 5 | 采购 1800 条，无标签 | 数据就绪度评估、小样本限制、规则引擎优先 |

断言基于推理步骤（"确认业务目标"、"评估数据就绪度"、"识别不平衡"），而非算法名称，同时包含反面断言（"不应直接推荐纯监督"）。
