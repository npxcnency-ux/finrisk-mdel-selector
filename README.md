# finrisk-model-selector

财务风控场景的 ML 算法选型 Skill，适用于 Claude Code。

## 解决什么问题

财务风控建模的核心矛盾是 **预测效果 vs 可解释性 vs 实施成本**。面对"用什么模型"这个问题，选错路径的代价很高：

- 异常率 < 1% 时直接上 XGBoost → 模型全判正常，Recall 接近 0
- 用未确认的"疑似"标签训练 → 标签偏差被模型放大
- 追求 Recall 不考虑审核容量 → 审核员疲劳，模型形同虚设
- 随机 split 数据集 → 时间泄露，回测完美上线骤降
- 特征只有 3-4 个就硬上复杂模型 → 不如简单规则引擎

本 Skill 提供一套 **Step 0-3 决策框架**，从业务目标出发，经数据就绪度、标签状态、数据量与不平衡联合评估，给出结构化的选型建议。当监管要求书面解释时，采用双轨架构（效果轨 + 解释轨）而非牺牲性能。

## 适用范围

**覆盖**：表格数据的批量异常检测 / 风险评分 — 费用报销审核、差旅异常、供应商付款风控、反欺诈稽核等。

**不覆盖**：实时流处理（Flink/Kafka Streams）、图像类欺诈检测（发票 OCR）、NLP 合同审计、信贷授信评分卡（A/B/C 卡体系）。

## 选型流程（Step 0-3）

```
Step 0: 业务目标与运营约束 → 费用类型澄清、误报/漏报权衡、审核容量、可解释性受众、监管要求（单轨/双轨）
Step 1: 数据就绪度评估 → 特征数量/质量、缺失率、时间字段可靠性、数据时间跨度
Step 2: 标签状态 → 有标签 / 弱标签 / 无标签，决定监督/半监督/无监督路径
Step 3: 数据量与不平衡联合评估 → 有效异常样本数决定模型复杂度上限
```

## 输出格式

每次推荐包含 7 个组件：

1. **业务上下文与约束确认** — 费用类型、误报/漏报权衡、运营容量、可解释性受众、监管要求、单轨/双轨部署模式
2. **数据就绪度评估** — 特征数量/质量、缺失率、时间字段、数据时间跨度、高基数特征
3. **推荐算法** — 主推 + 备选及选择理由；双轨时分别说明效果轨和解释轨模型
4. **不平衡处理方案** — 根据异常比例 + 有效异常样本数联合选策略
5. **评估指标** — PR-AUC / Precision@K / Recall，说明为什么
6. **双轨一致性校验方案**（仅双轨时）— Top-K 重叠率标准、差异样本审查流程
7. **下一步行动** — 数据准备 → 基线 → 迭代路径

## 关键设计决策

| 决策 | 理由 |
|------|------|
| Step 0 业务目标放首位 | 误报/漏报权衡决定评估指标，运营容量决定阈值天花板，可解释性需求决定单轨/双轨，最后考虑会导致推倒重来 |
| Step 0 先澄清费用类型 | 差旅/采购/招待的异常定义、合理阈值、标签质量差异极大，混为一谈导致特征工程和标签定义错位 |
| 双轨架构替代白名单 | 监管需书面解释时，效果轨（最优模型）+ 解释轨（评分卡/EBM）分别最优化，不互相妥协，Top-K 重叠率 ≥ 70% 校验一致性 |
| Step 1 数据就绪度评估 | 衍生后特征 < 5 个或缺失率 > 50% 时，ML 模型不如规则引擎，先评估避免浪费 |
| Step 3 合并数据量与不平衡 | 有效异常样本数（总量 × 异常率）才是真正的决策变量；< 200 条时监督模型难以稳定 |
| 异常率 < 1% 不直接纯监督 | 核心反直觉规则：先 Isolation Forest 扩召回，再监督精排 |
| 异常率 > 20% 先查标签定义 | 费用风控中超高异常率通常意味着标签过宽（如"超均值 1.5 倍"），需先确认定义合理性 |
| 有效异常样本 < 200 不直接上监督 | 绝对数量不足时监督模型交叉验证方差极大，优先半监督或浅层模型 |
| 三层生产架构（规则 → 无监督 → 有监督） | 支持冷启动逐步演进；训练集排除规则已命中样本，让模型学习规则遗漏的模式 |
| 运营容量先于指标优化 | 先问"每天能审多少条"再定阈值，避免预警淹没审核团队 |

## 触发条件

当用户提到以下关键词或描述相关场景时自动触发：

- 中文：选什么模型、用什么算法、有/没有标签、数据不平衡、需要可解释、财务风控建模、异常检测建模、评分卡、冷启动、报销审核、费控、费用管控、反洗钱、供应商风险、关联交易、合规模型、风险评分模型、报销异常
- 英文：model selection, fraud detection, expense anomaly, imbalanced dataset, anomaly detection, risk scoring, scorecard, AML, cold start, compliance model, vendor risk, expense review

## 评估

`evals/evals.json` 包含 8 个测试场景，覆盖：

| # | 场景 | 核心考察点 |
|---|------|-----------|
| 1 | 差旅报销 8 万条，异常 0.75% | 业务目标确认、严重不平衡两阶段策略、PR-AUC 选择 |
| 2 | 供应商付款 15 万条，无标签 | 业务目标澄清、数据就绪度评估、无监督路径、高基数特征、运营容量 |
| 3 | 信用卡 50 万条，双受众解释需求 | 业务目标确认、区分两类可解释性、不能只靠 SHAP、运营容量、时间切分 |
| 4 | 理赔 2 万条，180 条疑似标签 | 业务目标关联、弱标签识别、半监督路径 |
| 5 | 采购 1800 条，无标签 | 数据就绪度评估、小样本限制、规则引擎优先 |
| 6 | 贸易公司 5000 条，仅 3 个字段 | 数据就绪度红线、衍生特征空间分析、规则引擎优先 |
| 7 | 费用报销 10 万条，异常率 25% | 质疑标签定义、确认后不做不平衡处理 |
| 8 | 信用卡 3 万条，纯内部无解释需求 | 确认单轨部署、中度不平衡处理、效果优先模型 |

断言基于推理步骤（"确认业务目标"、"评估数据就绪度"、"识别不平衡"），而非算法名称，同时包含反面断言（"不应直接推荐纯监督"）。
