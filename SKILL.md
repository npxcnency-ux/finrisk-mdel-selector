# 财务风控 ML 模型选择指南

算法选型的核心矛盾：**预测效果 vs 可解释性 vs 实施成本**。财务风控场景有监管和审计压力，可解释性往往是硬约束，必须在选型最初就锁定，再通过业务指标反推应该选择什么技术指标来衡量模型效果。

> **适用范围**：本 skill 聚焦于**表格数据的批量异常检测/风险评分**场景。实时流处理（Flink/Kafka Streams 架构选型）、图像类欺诈检测（发票 OCR/证件伪造）、NLP 合同审计、信贷授信评分卡（A/B/C 卡体系）不在覆盖范围内。

**参考文件：**
- `references/algorithm-matrix.md` — 算法推荐矩阵、不平衡处理策略、评估指标选择
- `references/pitfalls.md` — 踩坑详解（标签偏差、概念漂移、时间泄露、数据就绪度等）
- `workflow.mmd` — 完整选型流程图（Mermaid 格式），可用于向用户展示决策路径

## 四步选型

> **信息不足时先提问**：如果用户未提供业务目标、数据概况、标签状态、异常比例或数据量，先提问再推荐。猜测然后给错误路径，代价远高于多问一句。使用AskUserQuestion向用户提问。

### Step 0：业务上下文与约束

**先问业务，再谈技术。** 这一步决定后续所有选择的方向、边界和部署模式（单轨/双轨）。

**前置澄清**：先确认具体的费用类型或业务场景（差旅报销 / 采购付款 / 招待费 / 关联交易 / 供应商回扣等）。不同费用类型的异常定义、合理阈值、标签质量差异极大——差旅报销的"异常"可能是超标准，采购付款的"异常"可能是虚假供应商，混为一谈会导致特征工程和标签定义错位。

需要明确的四个问题：
1. **误报和漏报哪个代价更高？**
   - 费用报销（人工跟进）→ 误报成本高 → 优化 Precision，调高阈值
   - 反欺诈/合规稽核 → 漏报损失大 → 优化 Recall，调低阈值
2. **审核团队每天能处理多少条？**
   - 这个数字直接决定模型阈值上限。Recall 再高，2000 条预警只能处理 200 条，模型等于无效
   - 如果用户说不清，给一个估算公式：每条审核耗时 × 审核人数 × 工作时长 = 日处理量
3. **模型输出需要解释给谁看？**（内部使用场景）
   - **内部团队（风控/合规）看排序** → 效果优先，XGBoost + SHAP summary 即可满足
   - **审核人员（运营）逐条判断** → 需要单条级别的风险归因，XGBoost + SHAP top-3 因素 或规则命中说明均可
   - **向被标记员工解释原因** → 强解释性，必须用评分卡或规则输出（"单笔金额超部门均值 3 倍"），SHAP waterfall 图对非技术人员不够直观
   - **多类受众共存** → 共享特征工程，按最强解释性需求决定是否需要额外的解释模型
   - **纯内部预测，无解释需求** → 无限制
4. **是否需要出具正式审计报告或接受外部监管检查？**（外部合规义务）
   - 有 → 可解释性是硬约束，**启用双轨架构**：效果轨选最优模型（XGBoost/GBDT+LR 等，不受限），解释轨选可审计模型（评分卡/EBM/决策树深度≤5），两轨共享特征工程，需做一致性校验（Top-K 重叠率）
   - 无 → 单轨即可，最优模型 + SHAP 辅助解释

**可解释性为什么放在 Step 0？** 在集团财务领域，合规性和可审计性带来的可解释性需求是硬约束，不是技术后置步骤。它决定了是单轨还是双轨部署——如果监管要求书面解释，必须额外部署一条解释轨，而不是牺牲效果轨的性能。

> **为什么用双轨而不是白名单？** 白名单机制会把效果模型限制在可解释模型范围内，导致性能天花板下降。双轨架构的核心思路：**效果和解释是两个独立目标，用两个模型分别最优化，而不是用一个模型同时妥协**。效果轨负责排序和预测，解释轨负责生成监管可接受的书面归因。两轨共享特征工程、共用训练数据，通过 Top-K 重叠率校验一致性（重叠率 < 70% 时需排查特征或数据问题）。

> **解释轨推荐模型**：评分卡（WOE + LR）适合需要书面评分报告的场景；EBM（GA²M，InterpretML）是玻璃盒模型，性能接近 XGBoost，每个特征有独立 shape function 可审计；决策树（深度 ≤ 5）适合需要导出 if-else 规则的场景。

**为什么 Step 0 而不是最后？** 业务目标决定评估指标（PR-AUC vs Precision@K vs Recall 底线），运营容量决定阈值天花板，可解释性需求决定单轨/双轨部署模式。如果最后才考虑，前面所有选择可能推倒重来。

### Step 1：数据就绪度评估

**数据质量决定模型复杂度上限。** 特征不够或质量太差时，再好的算法也跑不出结果。

| 检查项 | 红线 | 触发动作 |
|--------|------|---------|
| 可用特征数量 | 原始字段经特征工程衍生后仍 < 5 个有效特征 | 规则引擎优先，暂不建模 |
| 缺失率 | 关键字段缺失 > 50% | 先治理数据，或排除该字段 |
| 时间字段可靠性 | 无可靠时间戳 | 无法按时间切分，回测结果不可信，必须先解决 |
| 数据时间跨度 | < 一个完整业务周期（通常 < 12 个月） | 模型无法学到季节性模式，需特别说明局限性或等数据积累 |
| 类别特征基数 | 唯一值 > 1000（如收款方名称） | 需特殊编码，详见 `references/pitfalls.md` 高基数类别特征章节 |
| 特征稀疏度 | 某特征 > 90% 为同一值 | 该特征区分度极低，考虑丢弃 |

**关于特征数量的判定**：不要只看原始字段数。4 个原始字段（金额、日期、部门、供应商）经过特征工程可以衍生出 10+ 有效特征（金额偏离部门均值倍数、近 30 天消费频次、供应商集中度、是否周末提交等）。红线卡的是衍生后的有效特征数，不是原始字段数。如果用户说"只有 4 个字段"，先评估衍生空间再下结论。

**数据就绪度不过关的处理：**
- 衍生后特征仍 < 5 且缺失率高 → **只用规则引擎**，不要硬上模型。规则引擎不需要特征工程，维护成本低，效果可控
- 特征 5-15 个、质量尚可 → 可以建模，但限制模型复杂度（树深度 ≤ 5，不用深度学习）
- 特征 > 15 个、质量好 → 无限制，按后续步骤选型

### Step 2：标签状态

标签状态决定监督/半监督/无监督路径：

- **有审核确认标签** → 监督学习路径，继续 Step 3
- **只有"疑似"未经二次确认** → 弱标签，**跳至 `references/algorithm-matrix.md` 半监督路径**；核心原则：先评估标签质量，再决定是否复用
  - **快速指引**：用 Isolation Forest 对全量数据打异常分，检查这批弱标签是否集中在高异常分区域。若集中 → 标签有一定可信度，可用 Self-Training 迭代扩充；若分散 → 标签质量存疑，优先走无监督路径，同时安排人工抽样验证。弱标签数量 < 200 条时不建议直接做监督训练，噪声风险过高。
- **无任何标注** → 无监督路径，Step 3 中不平衡部分跳过，仅评估数据量

### Step 3：数据量与不平衡联合评估

**核心洞察**：真正决定模型可选范围的不是总数据量，也不是异常率，而是**有效异常样本数 = 总量 × 异常率**。8 万条 × 0.75% = 600 条异常样本，和 2 万条 × 3% = 600 条异常样本，面临的建模挑战本质相同。

#### 不平衡判定与处理（有标签时）

不平衡处理不能只看比例——**有效异常样本数 = 总量 × 异常率**才是真正决定建模难度的变量。10 万 × 0.8% = 800 条异常样本，建模可行；3000 × 3% = 90 条异常样本，比例更高但样本太少，模型难以学到稳定模式。

| 异常比例 | 有效异常样本数 | 判定 | 处理策略 |
|---------|--------------|------|---------|
| **> 20%** | **—** | **先检查标签定义** | **费用风控中 > 20% 异常率本身是强业务信号——先确认标签定义是否过于宽松（如"超均值 1.5 倍"把正常波动也标为异常）。确认标签合理后无需特殊处理** |
| 5%–20% | > 500 | 轻度不平衡 | 仅需 `class_weight='balanced'`，按数据量选模型 |
| 5%–20% | < 500 | 轻度不平衡但样本不足 | `class_weight='balanced'` + 限制模型复杂度（树深度 ≤ 5）|
| 1%–5% | > 500 | 中度不平衡 | SMOTE 过采样 + class_weight，按数据量选模型 |
| 1%–5% | < 500 | 中度不平衡且样本不足 | SMOTE + class_weight + 限制模型复杂度，考虑半监督路径补充 |
| **< 1%** | **任意** | **严重不平衡** | **Isolation Forest 初筛扩召回 → 监督精排**（见下方关键规则） |

**关键反直觉规则**：异常率 < 1% 时，即使有标签也**不要直接用纯监督模型**。模型会偏向全判正常，Recall 接近 0。正确策略：**Isolation Forest 初筛扩召回 → 监督精排**（即使你有 600 条标注异常也如此）。

**有效异常样本数的底线**：当有效异常样本 < 200 条时，无论异常比例多高，监督模型都难以学到稳定的异常模式。此时应优先考虑半监督路径或限制模型复杂度（评分卡/浅层决策树）。

#### 数据量与模型选择矩阵

| 数据量 | 有效异常样本 | 有标签推荐（效果轨） | 无监督推荐 |
|--------|------------|---------------------|-----------|
| < 5,000 条 | < 250 | 评分卡（WOE + LR）或决策树（深度≤5） | Isolation Forest + LOF |
| 5,000 ~ 10 万 | 250 ~ 5,000 | XGBoost + SHAP | Isolation Forest + LOF |
| > 10 万 | > 5,000 | GBDT + LR 两阶段 | Autoencoder（重构误差） |

**注意**：此处数据量指可用于建模的记录总数（含正常 + 异常），不是异常样本数。有效异常样本数用于辅助判断模型能否学到足够模式。

**双轨架构下的选择**：当 Step 0 启用双轨架构时，上表"有标签推荐"列即为效果轨模型。解释轨模型另选：< 5K 用评分卡或决策树（深度≤5），≥ 5K 优先用 **EBM（GA²M）**（性能最接近效果轨，且完全可审计），备选评分卡。两轨共享特征工程，通过 Top-K 重叠率校验一致性。

**严重不平衡（< 1%）时的组合规则**：无论数据量多大，第一步都是 Isolation Forest 初筛扩召回，**精排模型**根据数据量选择：< 5K 用评分卡/决策树（深度≤5），5K-10 万用 XGBoost，> 10 万用 GBDT+LR。双轨架构下，精排即为效果轨，解释轨同样在初筛候选集上训练。

详见 `references/algorithm-matrix.md` 的完整推荐矩阵。

## 三层生产架构

```
第一层（规则引擎）— 零训练成本，立即可用
  ↓ 明确违规直接送人工审核队列，不再进入第二层

第二层（无监督初筛）— 冷启动友好，不依赖标签
  ↓ Isolation Forest + LOF 对剩余记录打分，高风险候选集送第三层

第三层（有监督精排）— 需要标签，后期补充
  ├─ 效果轨：按数据量选最优模型（< 5K: 评分卡 / 5K-10万: XGBoost / > 10万: GBDT+LR）
  ├─ 解释轨（仅双轨时）：评分卡 或 EBM，生成监管可接受的书面归因
  └─ 输出风险概率 + 审核优先级列表（双轨时附带解释报告）
```

三层阈值协调原则：先确认审核团队每日处理量（Step 0），从第三层往上倒推候选集大小，而不是模型说多少就送多少。冷启动时第三层缺席：规则 + 无监督先跑，人工审核结果积累标签后再补充监督模型。

**双轨一致性校验**：效果轨和解释轨的 Top-K 排序结果做重叠率检查。重叠率 ≥ 70% 视为一致；< 70% 时需排查：是否有特征在一个模型中被压缩/丢弃，或某类异常只被一个模型捕捉。不一致本身不一定是问题——两个模型视角不同是正常的——但需要人工审查差异样本，确认效果轨的预测在解释轨中有合理归因。

## 关键 Gotchas

以下是 Claude 在没有此 Skill 时最容易犯的错误：

**1. 有标签 = 直接用纯监督**
异常率 < 1% 时强行 XGBoost，模型学会了"全判正常"，Accuracy 99%，Recall 接近 0。正确做法见 Step 3。

**2. 用 ROC-AUC 评估极不平衡模型**
异常率 0.5% 时，全判正常的模型 ROC-AUC 也能 0.9+。应改用 **PR-AUC**，它对稀有类别更敏感。

**3. 随机 split 训练/测试集**
统计类特征（如"员工历史异常率"）会把未来数据泄露到训练集，回测完美但上线骤降。必须按时间切分。

**4. 追求 Recall 忽略运营容量**
Recall 95% 但每天 3000 条预警 → 审核员疲劳 → 拒绝使用模型。Step 0 先对齐团队处理量再定阈值。

**5. 模型上线后不监控**
欺诈者会学习并规避模型规律，6 个月后效果静默下降。每季度监控 PSI，> 0.25 触发重训。

**6. 评分卡/LR 忽略多重共线性**
金额类字段高度相关时，逻辑回归系数会出现方向翻转，评分卡输出无法解释。必须做 VIF 检测，VIF > 5 需处理。详见 `references/pitfalls.md` 多重共线性章节。

**7. 高基数类别特征直接 One-Hot**
收款方名称等字段有上万唯一值，One-Hot 导致维度爆炸；Target Encoding 不在 K-Fold 内做会造成严重数据泄露。详见 `references/pitfalls.md` 高基数类别特征章节。

**8. 数据就绪度不足硬上模型**
特征只有 3-4 个、缺失率高、无可靠时间字段，强行训练复杂模型。结果不如简单规则引擎，还浪费大量工程资源。Step 1 先评估数据就绪度。

**9. 规则引擎与模型边界模糊**
三层架构中，规则引擎（第一层）已经拦截的明确违规记录，如果不加处理地放入监督模型（第三层）的训练集，模型会学到"规则已经覆盖的模式"而不是"规则遗漏的模式"。后果：模型上线后与规则引擎高度重叠，增量价值极低。正确做法：**训练集中排除规则引擎已命中的样本**，让模型专注学习规则无法捕捉的复杂异常模式。评估时也要在规则引擎未命中的样本上评估，而不是全量评估（否则指标虚高）。

**10. 有效异常样本 < 200 直接上监督模型**
有效异常样本数（总量 × 异常率）不足 200 条时，即便异常比例看起来不低（如 5000 条 × 3% = 150 条），监督模型也学不到稳定模式，交叉验证方差极大。此时应优先走半监督路径或限制模型复杂度（评分卡/浅层决策树深度 ≤ 3），而不是默认用 XGBoost。很多人只看比例不算绝对数量，这是一个常见的判断盲区。

## 输出格式

给出选型建议时包含：
1. **业务上下文与约束确认** — 误报/漏报权衡、运营容量、可解释性需求（解释给谁看）、监管要求、单轨/双轨部署模式
2. **数据就绪度评估** — 特征数量/质量、缺失率、时间字段、高基数特征
3. **推荐算法**（主推 + 备选）及选择理由；双轨时分别说明效果轨和解释轨模型
4. **不平衡处理方案**（如异常率 < 5%）
5. **评估指标**（PR-AUC / Precision@K / Recall，说明为什么）
6. **双轨一致性校验方案**（仅双轨时）— Top-K 重叠率标准、差异样本审查流程；**单轨时标注"不适用"并简要说明原因**
7. **下一步行动**（数据准备 → 基线 → 迭代路径）

完整输出示例见 `examples/sample-recommendation.md`。
