# 财务风控 ML 模型选择指南

算法选型的核心矛盾：**预测效果 vs 可解释性 vs 实施成本**。财务风控场景有监管和审计压力，可解释性往往是硬约束，要通过业务指标反推出应该选择什么技术指标来衡量模型效果。

> **适用范围**：本 skill 聚焦于**表格数据的批量异常检测/风险评分**场景。实时流处理（Flink/Kafka Streams 架构选型）、图像类欺诈检测（发票 OCR/证件伪造）、NLP 合同审计、信贷授信评分卡（A/B/C 卡体系）不在覆盖范围内。

**参考文件：**
- `references/algorithm-matrix.md` — 算法推荐矩阵、不平衡处理策略、评估指标选择
- `references/pitfalls.md` — 踩坑详解（标签偏差、概念漂移、时间泄露等）

## 五步选型

> **信息不足时先提问**：如果用户未提供标签状态、异常比例或数据量，先提问再推荐。猜测比例然后给错误路径，代价远高于多问一句。

### Step 1：标签状态
- **有审核确认标签** → 监督学习路径，继续 Step 2
- **只有"疑似"未经二次确认** → 弱标签，**跳至 `references/algorithm-matrix.md` 半监督路径**；核心原则：先评估标签质量，再决定是否复用
- **无任何标注** → 无监督路径，Step 2 跳过，继续 Step 3

### Step 2：标签平衡性（有标签时）

| 异常比例 | 判定 | 处理概要 |
|---------|------|---------|
| 5%–20% | 轻度不平衡 | 仅需 `class_weight='balanced'`，直接走 Step 3 |
| 1%–5% | 中度不平衡 | SMOTE 过采样 + class_weight，直接走 Step 3 |
| **< 1%** | **严重不平衡** | **Isolation Forest 初筛扩召回 → 监督精排**（见下方） |

**关键反直觉**：异常率 < 1% 时，即使有标签也**不要直接用纯监督模型**。模型会偏向全判正常，Recall 接近 0。正确策略：**Isolation Forest 初筛扩召回 → 监督精排**（即使你有 600 条标注异常也如此）。

详见 `references/algorithm-matrix.md` 的不平衡处理策略和完整处理矩阵。

### Step 3：可用数据量

数据量直接决定监督模型选择和无监督模型的上限：

| 数据量 | 有标签后推荐 | 无监督推荐 | 原因 |
|--------|-------------|-----------|------|
| < 5,000 条 | 评分卡（WOE + LR）或决策树（深度≤5） | Isolation Forest + LOF | 小样本下树模型易过拟合，LR 更稳定 |
| 5,000 ~ 10 万 | XGBoost + SHAP | Isolation Forest + LOF | 样本足够支撑树模型复杂度 |
| > 10 万 | GBDT + LR 两阶段 | Autoencoder（重构误差） | 大样本下 Autoencoder 能学好正常模式，两阶段概率校准更优 |

**注意**：此处数据量指可用于建模的记录总数（含正常 + 异常），不是异常样本数。

**Step 2 + Step 3 组合规则**（严重不平衡 < 1% 时）：无论数据量多大，第一步都是 Isolation Forest 初筛扩召回，**精排模型**根据数据量选择：< 5K 用评分卡/决策树，5K-10 万用 XGBoost，> 10 万用 GBDT+LR。不平衡度 ≥ 1% 时，Step 2 的处理策略独立于 Step 3，按各自推荐执行即可。

### Step 4：可解释性要求

- **监管/审计硬要求** → 评分卡（WOE + LR）或决策树，需要能出具书面解释
- **内部风控，辅助说明** → XGBoost + SHAP
- **纯预测，无解释需求** → 任意模型

**区分两类受众**：给合规团队看排序（效果优先）≠ 向员工解释被标记原因（强解释性）。两个需求应分别用不同模型，共享特征工程。

### Step 5：业务价值权衡

先问用户：**"误报和漏报哪个代价更高？"**

| 场景 | 更怕什么 | 优化方向 |
|------|---------|---------|
| 费用报销（人工跟进）| 误报多 → 审核成本高 | 提高 Precision，调高阈值 |
| 反欺诈/合规稽核 | 漏报多 → 损失大 | 提高 Recall，调低阈值 |

**运营可行性先于指标优化**：先问"审核团队每天能处理多少条"，再反推阈值。Recall 再高，2000 条预警只能处理 200 条，模型等于无效。

## 三层生产架构

```
第一层（规则引擎）— 零训练成本，立即可用
  ↓ 明确违规直接送人工审核队列，不再进入第二层

第二层（无监督初筛）— 冷启动友好，不依赖标签
  ↓ Isolation Forest + LOF 对剩余记录打分，高风险候选集送第三层

第三层（有监督精排）— 需要标签，后期补充
  ↓ 小样本(< 5K): 评分卡(WOE+LR) / 中样本(5K-10万): XGBoost / 大样本(> 10万): GBDT+LR
  ↓ 输出风险概率，按概率降序生成审核优先级列表
```

三层阈值协调原则：先确认审核团队每日处理量，从第三层往上倒推候选集大小，而不是模型说多少就送多少。冷启动时第三层缺席：规则 + 无监督先跑，人工审核结果积累标签后再补充监督模型。

## 关键 Gotchas

以下是 Claude 在没有此 Skill 时最容易犯的错误：

**1. 有标签 = 直接用纯监督**  
异常率 < 1% 时强行 XGBoost，模型学会了"全判正常"，Accuracy 99%，Recall 接近 0。正确做法见 Step 2。

**2. 用 ROC-AUC 评估极不平衡模型**  
异常率 0.5% 时，全判正常的模型 ROC-AUC 也能 0.9+。应改用 **PR-AUC**，它对稀有类别更敏感。

**3. 随机 split 训练/测试集**  
统计类特征（如"员工历史异常率"）会把未来数据泄露到训练集，回测完美但上线骤降。必须按时间切分。

**4. 追求 Recall 忽略运营容量**  
Recall 95% 但每天 3000 条预警 → 审核员疲劳 → 拒绝使用模型。先对齐团队处理量再定阈值。

**5. 模型上线后不监控**  
欺诈者会学习并规避模型规律，6 个月后效果静默下降。每季度监控 PSI，> 0.25 触发重训。

**6. 评分卡/LR 忽略多重共线性**  
金额类字段高度相关时，逻辑回归系数会出现方向翻转，评分卡输出无法解释。必须做 VIF 检测，VIF > 5 需处理。详见 `references/pitfalls.md` 多重共线性章节。

**7. 高基数类别特征直接 One-Hot**  
收款方名称等字段有上万唯一值，One-Hot 导致维度爆炸；Target Encoding 不在 K-Fold 内做会造成严重数据泄露。详见 `references/pitfalls.md` 高基数类别特征章节。

## 输出格式

给出选型建议时包含：
1. **推荐算法**（主推 + 备选）及选择理由
2. **不平衡处理方案**（如异常率 < 5%）
3. **解释性方案**（规则 / 决策树 / SHAP，对应谁看）
4. **评估指标**（PR-AUC / Precision@K / Recall，说明为什么）
5. **下一步行动**（数据准备 → 基线 → 迭代路径）

完整输出示例见 `examples/sample-recommendation.md`。
