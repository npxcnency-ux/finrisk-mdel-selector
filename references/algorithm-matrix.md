# 算法推荐参考矩阵

## 有标签路径

| 条件 | 推荐算法 | 原因 |
|------|---------|------|
| 样本 < 5000 + 强解释性（监管场景）| 评分卡（WOE/IV + 逻辑回归）| 系数可直接出具书面解释，满足银行/保险监管要求 |
| 样本 < 5000 + 规则需求 | 决策树（深度≤5） | 可导出 if-else 规则 |
| 样本 > 5000 + 中等解释性 | XGBoost + SHAP | 效果强，SHAP 辅助解释 |
| 样本 > 5000 + 强解释性（监管场景）| **EBM（GA²M，InterpretML）** | 玻璃盒模型，性能接近 XGBoost，每个特征有独立 shape function 可审计；双轨架构下作为解释轨首选 |
| 样本 > 5000 + 高解释性 | LightGBM + 规则提取 | 速度快，可配合规则层 |
| 样本 > 10 万 + 中等解释性 | GBDT + LR 两阶段 | 树做特征交叉，LR 做可解释输出，概率校准更好 |
| 样本 > 10 万 + 强解释性（监管场景）| **EBM（GA²M）** | 双轨架构解释轨首选，大数据量下优势更明显 |
| 时序消费行为 | LSTM + 注意力机制 | 捕捉序列中的异常模式 |
| **严重不平衡（< 1%）** | **Isolation Forest 初筛 → XGBoost 精排**（双轨时解释轨用 EBM）| **即使有标签也要先扩召回，再精排（Step 3 关键规则）** |

## 弱标签/半监督路径

适用场景：有少量经确认标签（< 500 条），或标签质量存疑（"疑似"未经二次核实）。Step 2 判定为弱标签时跳转至此。

| 条件 | 推荐方法 | 原因 |
|------|---------|------|
| 有少量确认标签（< 500 条）| Label Propagation / Label Spreading | 利用有标签样本向无标签样本传播，适合特征空间连续的场景 |
| 标签质量不稳定 | Self-Training（自训练）| 用高置信度预测结果扩充标签集，迭代训练 |
| 有标签 + 无标签数据量大 | Semi-Supervised Isolation Forest | 将已知异常作为约束引导无监督初筛方向 |
| 先无监督、后有监督 | **Isolation Forest 初筛 → 人工审核 → XGBoost 精排** | 冷启动最实用路径，无需等到足够标签再启动 |

**注意**：弱标签场景往往比直接走无监督更容易被忽视。如果用户说"有几百条疑似记录但未经确认"，不要直接上 XGBoost，先评估标签质量。

## 无/弱标签路径（完全无标注）

| 条件 | 推荐算法 | 原因 |
|------|-----------|------|
| 单维异常（金额/频次）| IQR / Z-Score 规则 | 可解释，易维护 |
| 多维异常 + 需要解释 | K-Means 分群 → 离群分析 | 结果可对业务解释 |
| 多维异常 + 效果优先 | Isolation Forest | 财务场景最常用，快速高效 |
| 局部密度异常（拆单等）| LOF（Local Outlier Factor）| 对拆单、频繁小额敏感 |
| 特征高维或有交叉关系 | Autoencoder（重构误差）| 学习正常模式，偏差即异常；需数据量 > 10 万 |
| 时序消费 + 无标签 | LSTM Autoencoder | 捕捉时序偏离正常序列 |

## 数据就绪度对算法选择的影响

Step 1 数据就绪度评估直接约束可选算法范围：

| 数据就绪度 | 可选算法范围 | 不可选 |
|-----------|-------------|-------|
| 衍生后特征 < 5 个 | 规则引擎（IQR/Z-Score/业务规则）| 任何 ML 模型 |
| 衍生后特征 5-15 个、质量尚可 | 评分卡、决策树（深度≤5）、EBM、Isolation Forest、LOF | Autoencoder、深度学习 |
| 衍生后特征 > 15 个、质量好 | 无限制，按 Step 3 数据量选择 | — |
| 关键字段缺失 > 50% | 先做数据治理，治理后重新评估 | 所有模型 |
| 数据时间跨度 < 12 个月 | 可建模但需说明季节性局限性 | 依赖长周期模式的特征（如同比指标） |
| 无可靠时间字段 | 可建模但回测不可信，需特别说明局限性 | 依赖时序切分的评估方案 |

## 标签不平衡处理策略

不平衡处理需同时考虑异常比例和有效异常样本数（= 总量 × 异常率）。

| 异常比例 | 有效异常样本数 | 判定 | 处理策略 |
|---------|--------------|------|---------|
| > 20% | — | **先检查标签定义** | 费用风控中 > 20% 异常率本身是强业务信号，需先确认标签定义是否过于宽松（如"超均值 1.5 倍"是否合理）。确认标签合理后无需特殊处理 |
| 5%–20% | > 500 | 轻度不平衡 | 调整 `class_weight='balanced'` |
| 5%–20% | < 500 | 轻度不平衡但样本不足 | `class_weight='balanced'` + 限制模型复杂度 |
| 1%–5% | > 500 | 中度不平衡 | SMOTE 过采样 + class_weight |
| 1%–5% | < 500 | 中度不平衡且样本不足 | SMOTE + class_weight + 限制复杂度，考虑半监督路径 |
| < 1% | 任意 | 严重不平衡 | Isolation Forest 初筛 → 监督精排 |

## 评估指标选择

指标选择应与 Step 0 确定的业务目标对齐：

| 场景 | 主要指标 | 原因 |
|------|---------|------|
| 极不平衡（< 1%）| PR-AUC | ROC-AUC 在此场景虚高，全判正常也能 0.9+ |
| 中度不平衡（1%-5%）| PR-AUC（主）+ AUC-ROC（辅） | 1%-5% 是灰区：ROC-AUC 仍有一定虚高风险，PR-AUC 更敏感；但 ROC-AUC 作为辅助参考仍有价值，尤其在模型间对比时 |
| 轻度不平衡或均衡（> 5%）| AUC-ROC | 不受阈值影响，反映整体区分能力，在此比例下不会虚高 |
| 运营阈值决策 | Precision@K | K = Step 0 确认的团队每日处理量，关注前 K 条命中率 |
| 合规漏报代价大 | Recall | 设定最低召回率底线（如 80%） |
| 综合评估 | F1 | Precision 和 Recall 的调和平均 |

**绝对不要用 Accuracy**：异常率 0.5% 时，全判正常也有 99.5% 准确率。
