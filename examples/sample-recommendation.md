# 输出示例：差旅报销异常检测选型建议

> 场景：某企业 8 万条差旅报销记录，经审核确认异常 600 条（0.75%），审计部门要求可解释，审核团队每天能处理约 50 条。

---

## 1. 业务上下文与约束确认

- **误报 vs 漏报**：差旅报销场景，误报会产生审核成本，但漏报有审计风险。综合来看，需要在合理 Precision 下保证 Recall 底线（≥ 70%）
- **运营容量**：审核团队每天处理 50 条 → 模型输出的每日预警量不应超过 50 条，否则预警积压、审核员疲劳
- **可解释性需求**：
  - **受众 1 — 内部团队（风控/合规）**：需要排查优先级列表和整体风险趋势，效果优先，XGBoost + SHAP summary 可满足
  - **受众 2 — 审核人员（运营）**：逐条判断是否真异常，需要单条级别的风险归因，XGBoost + SHAP top-3 因素 或规则命中说明
  - **受众 3 — 被标记员工**：需要理解为什么被标记，强解释性，必须用评分卡规则输出（"单笔金额超过部门均值 3 倍"、"30 天内同一商户消费 5 次"）
  - 三类受众共存 → 共享特征工程，合规+运营可共用 XGBoost + SHAP，员工解释需额外训练评分卡
- **外部合规义务**：审计部门要求输出结果能出具正式审计报告，解释清楚为什么某张单据有问题 → 可解释性是硬约束 → **启用双轨架构**
  - **效果轨**：选最优模型（不受可解释性限制），负责排序和预测
  - **解释轨**：选可审计模型，负责生成面向员工和审计的书面归因
  - 合规+运营共用效果轨输出（排查优先级 + SHAP top-3），员工解释使用解释轨输出（评分卡规则）

## 2. 数据就绪度评估

- **特征数量**：差旅报销数据通常包含 15-20 个可用字段（金额、目的地、交通方式、住宿天数、部门、审批链等），满足建模要求
- **缺失率**：需确认关键字段（金额、日期、部门）缺失率，假设 < 5%，可直接建模
- **时间字段**：有报销提交日期，可用于按时间切分训练/测试集
- **高基数特征**：如有商户名称字段（唯一值可能上千），需做 Target Encoding（K-Fold 内）或实体聚合，不可直接 One-Hot

## 3. 推荐算法（双轨架构）

**效果轨：Isolation Forest 初筛 + XGBoost 精排（两阶段）**

- 异常率 0.75%（< 1%），属于严重不平衡，纯监督模型会偏向全判正常
- 第一阶段：Isolation Forest 对全量 8 万条打异常分，扩大召回范围，筛出高风险候选集（如 Top 3000）
- 第二阶段：XGBoost 在候选集上用 600 条已确认标签精排，输出风险概率
- 用途：生成排查优先级列表（合规团队 + 审核运营人员使用）

**解释轨：评分卡（WOE + LR）**

- 在同一候选集上训练，共享特征工程
- 系数可直接映射为"因 XX 字段偏离正常范围导致评分下降 N 分"
- 用途：生成面向被标记员工的书面解释报告（"单笔金额超过部门均值 3 倍"、"30 天内同一商户消费 5 次"）

**备选解释轨模型**：如果评分卡性能差距过大，可换用 EBM（GA²M），性能更接近 XGBoost 且完全可审计。

## 4. 不平衡处理方案

- 异常率 0.75%，**不使用**纯 SMOTE 或 class_weight（不平衡程度太高，效果有限）
- 采用两阶段策略：Isolation Forest 先将 8 万条压缩为 ~3000 条高风险候选集，候选集内异常占比提升至 ~15%，再在此基础上训练 XGBoost
- XGBoost 训练时额外设置 `scale_pos_weight` 微调

## 5. 评估指标

| 指标 | 用途 | 原因 |
|------|------|------|
| **PR-AUC**（主指标） | 模型选型对比 | 0.75% 异常率下 ROC-AUC 虚高（全判正常也能 0.9+），PR-AUC 对稀有类更敏感 |
| **Precision@50** | 运营阈值决策 | 审核团队每天处理 50 条（Step 0 确认），关注前 50 条的命中率 |
| **Recall**（设底线 ≥ 70%） | 合规兜底 | 漏报有审计风险，需保证最低召回 |

**绝对不用 Accuracy**：0.75% 异常率下全判正常也有 99.25% 准确率。

## 6. 双轨一致性校验

- **校验方式**：取效果轨（XGBoost）和解释轨（评分卡）的 Top-50 输出，计算重叠率
- **合格标准**：重叠率 ≥ 70%（35/50 条重合）
- **不合格时排查方向**：
  - 某些特征在评分卡中被 WOE 分箱压缩了信息 → 调整分箱策略
  - 某类异常模式只被 XGBoost 捕捉（如特征交叉效应）→ 在评分卡中增加交叉特征
  - 差异样本人工审查，确认效果轨的预测在解释轨中有合理归因

## 7. 下一步行动

1. **数据准备**：按时间切分训练/测试集（如 2023 年前训练，2023 年后验证），**不要随机 split**——统计类特征（如"该员工历史异常率"）会造成时间泄露
2. **特征工程**：构造金额偏离度（vs 部门均值/个人历史）、消费频次、商户集中度、审批链长度异常等特征；注意 VIF 检查（金额类字段容易共线）
3. **基线模型**：先用纯规则引擎（超标准、重复发票）建立基线，再叠加 Isolation Forest，最后加效果轨（XGBoost）+ 解释轨（评分卡），做 Top-K 重叠率校验
4. **上线后监控**：每季度检查 PSI（> 0.25 触发重训），建立"模型没发现但人工发现了"的反向标注机制；双轨重叠率持续监控
